# プロジェクト定義: hakone-e2

## 1. プロジェクト名

hakone-e2（箱根駅伝 Emotional/EG-Space PoC）

## 2. 目的（Purpose）

- 箱根駅伝を題材にした **「感情を含んだ意味空間（EG-Space）」のミニ版** を構築する。
- 単なる順位・記録だけでなく、
  - 区間ごとのドラマ
  - 視聴者・ファンの感情の揺れ
  - 期待と裏切り・サプライズ
  などの情報を整理し、「どんな試合展開がどんな意味を持つか」を表現できるようにする。
- 将来的に、スポーツや他領域での **“感情を含んだプロジェクトマネジメント” や EG-Space 設計** にフィードバックするための PoC として位置づける。

## 3. スコープ（Scope）

- 対象:
  - 箱根駅伝（過去大会1〜2回分）を中心とした「試合展開＋感情」のモデル化。
  - 必要に応じて、実況・解説・SNS反応などのテキスト断片。
- 含めるもの:
  - 「試合展開イベント（例: 〇区での逆転）」と「意味・感情（例: 安心感／不安／期待／絶望）」の対応づけ。
  - vpm-mini と同様の C/G/δ / Next3 に基づく PM 的な整理（この PoC をどう進めるか）。
  - 将来の EG-Space / 5セル設計に還元できる形でのメタな振り返り（何がうまくいったか／難しかったか）。
- 含めないもの（現フェーズ）:
  - 全大会・全区間の完全なアーカイブ化。
  - 放送権や実データ利用に関する実務的な調整。
  - 商用プロダクトとしての完全な UX / UI 実装。

## 4. 成功条件（Success Criteria）

- hakone-e2 プロジェクトについて PM Kai（または専用PM）が、
  - 現在地（C）
  - ゴール（G）
  - ギャップ（δ）
  - Next 3（次のアクション）
  を一貫した形で説明できる。
- 少なくとも 1〜2 個の「試合展開イベント → 意味・感情の解釈」のペアが、
  - 文書として整理され、
  - 将来の EG-Space 設計に引用できる形で残っている。
- 「この PoC から vpm-mini / EG-Space 設計にフィードバックすべきポイント」が 2〜3 個以上、明文化されている。

## 5. 制約（Constraints）

- 実データ（映像・記録・テキスト）は、現時点では公開情報や個人的なメモ・印象ベースにとどめる。
- 箱根駅伝に関する感情・意味づけは、あくまで啓個人の観点をベースとし、一般論・公式見解とは切り離して扱う。
- 実装・システム面では、vpm-mini と同様に **GitHub + LLM** を基本とし、外部インフラへの依存は最小限とする。

## 6. フェーズ（現時点の位置づけ）

- 現在地（2025-11-23 時点）:
  - vpm-mini 側で PM Kai v1 のベースラインが整ったタイミングで、hakone-e2 を **「2つ目の PM 対象プロジェクト」** としてオンボードし始める段階。
  - まだデータや具体的なイベント整理はほとんど進んでおらず、「目的・スコープ・成功条件」と PM の器（STATE / weekly）を作るところ。
- 次のフェーズ候補:
  - シンプルな「1大会／数イベント」の意味付けを行い、C/G/δ / Next3 で進捗を管理できるようにする。
  - vpm-mini 側の EG-Space / 5セル議論と接続し、「スポーツ×感情」の整理からどんな設計指針が得られるか検討する。
