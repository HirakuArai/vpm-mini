vpm-mini プロジェクトビジョン / 背景（Draft）
1. プロジェクトの背景

このプロジェクトの出発点には、「啓個人に属人化している知的な仕事」の問題がある。

データ分析

意思決定支援

レポート作成や説明・相談の相手

といった仕事が、いまのところ啓に強く依存しており、啓が不在になると継続が難しい場面が多い。

しかし、この問題は啓だけの特殊事情ではなく、社会のあちこちで見られる「属人化した知的仕事」の縮図でもある。
vpm-mini は、まずこの個人的な文脈をテストベッドとして扱いながら、より一般化可能な解決パターンを探るためのプロジェクトである。

2. コア目的（普遍G）

このプロジェクトのコア目的（普遍G）は次の2点に集約される。

知的な仕事の「現在地C・目標G・差分δ」と、そのときの行動や結果（Evidence）を、STATE/docs/reports という SSOT に一元的に刻み込むこと。

いま何を目指しているのか（G）

いまどこにいるのか（C）

そこに至るまでに何が行われ、どんな結果になったのか（δとEvidence）

を、人間とAIがともに参照できる構造として残す。

人とAIが「小さく動く → 振り返る → 修正する」ループを、安心して何度も回せるようにすること。

精度よりベクトル（70%の確信度で一歩動く）

巻き戻し可能性（Evidence と Git の履歴でいつでも戻れる）

「動いた結果」も SSOT に蓄積し、次の判断の材料にする

この2つを満たすことで、属人的な判断や経験を「説明可能な構造」として外在化し、
人とAIの両方が参照できる「知的仕事の地図」を育てていく。

3. 現在のフォーカス（個人G＝テストベッド）

vpm-mini は、まず啓個人の仕事を題材にした テストベッド として設計されている。

まずは啓個人に属人化しているデータ分析・意思決定支援の仕事を、
「啓がいなくても回せる」レベルまで構造化・外在化することを主な適用対象とする。

その目的は、啓自身が「本人にしか向き合えない課題」（生き方や幸福の追求など）に
集中できる余白をつくることである。AIと仕組みの力を借りて、
うまくいく確率や判断の質を上げつつも、結果の成否に過度に自己評価を縛られず、
「本人にしかできない部分に、正直に向き合っていること自体」に
充実感や幸せを感じられる状態を目指す。

ここで得られたパターンや構造は、将来、他の人やチームにも再利用できる
汎用テンプレートになることを期待している。

この個人Gは、プロジェクトにとって単なる「使いやすいユースケース」ではなく、
何のために仕組みを作るのか（人生側の目的） を常に思い出させるアンカーでもある。

4. ドキュメント（SSOT）の役割と「二重ループ」

このプロジェクトでは、ドキュメント（STATE / docs / reports）を単なる記録ではなく、
「現在地CとゴールG、差分δを表現するための SSOT」 として扱う。

内容のループ（本来の開発・運用）

現実世界やシステムで何かが起こる

Evidence が reports/** に溜まる

STATE / docs が更新され、C/G/δがわずかに書き換わる

それを読みながら次のアクションや North Star が決まる

仕組みのループ（ドキュメント更新の AI 化）

「どうやってその更新が行われたか？」を観察する

/ask, u_contract, persist-report, codex-brief など、
更新パターンを抽象化して仕組みに落とす

次の更新は、前回よりも少しだけ「AI＋仕組み」で楽になる

1回のドキュメント更新は、

プロジェクトを前に進める一歩であり（内容のループ）

更新を自動化する仕組みを育てる一歩でもある（仕組みのループ）

という 二重の意味 を持つ。

5. フェーズ構成のダイジェスト

vpm-mini では、上記の目的を段階的に実現するために、
フェーズを大きくいくつかに分けて進めている。

Step-A〜Phase 1：
チャットUI・ログ・要約パイプライン・品質ガードを整備し、
Docker Compose + Redis + Prometheus + Grafana + ROUGE-L exporter で
「Evidence付きの小さな自律ループ」をPoCする。

Phase 2：
kind + Knative で Hello KService が READY=True になる足場を作り、
コンテナ化されたセルを Kubernetes 上で動かす基礎を整える。

Phase 3（現在地）：
小規模 Swarm の足場づくり。

metrics-echo レーン（監視セル）

persist-report レーン（レポート半自動整理セル）

codex レーン（codex-brief / codex-runner の仕様設計）

以降のフェーズでは、GKE Autopilot への移行や Biz KPI への適用などを通じて、
このテストベッドを実務に耐える「業務OS」へと拡張していく。

6. この文書の使い方

この文書は、次のような場面で参照されることを想定している。

新しいAIエージェントや人間の開発者が、vpm-mini に参加するとき

/ask や新しいセルが、「何を優先すべきか」を判断する文脈が必要なとき

プロジェクトの方向性に迷ったときに、「なぜ始めたのか」に立ち返るとき

原則として：

vpm-mini に関する新しいセッションや設計議論を始める際には、
本文書の内容（少なくとも 1〜3章）を前提として扱う。

背景や目的に大きな変更が入った場合は、必ず本ファイルを更新し、
STATE/current_state.md からリンクを貼る。

（Draft のため、Phase 3 以降の進捗に応じて適宜更新すること。）
